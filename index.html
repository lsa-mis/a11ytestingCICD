Automated accessibility testing in CI/CD

Accessibility,
enforced by CI

Every push and pull request renders the site in a real browser, audits it against WCAG, and fails the build if there's a violation.

  • Engine · Siteimprove Alfa
  • Browser · Playwright
  • CI · GitHub Actions

Accessibility regresses quietly

  • Audits are periodic. Code changes daily. A page passes in March and breaks in April.
  • Nothing in the pipeline enforces it. Tests, linting, and types are gated — accessibility isn't.
  • You find out last. Usually from a user, a complaint, or a compliance review.

The idea

A red check on the pull request the moment a WCAG violation is introduced.

It audits the rendered DOM — after CSS and JavaScript — so it sees what users actually get.

What it catches — and what it doesn't

✓ Caught automatically

  • Missing image alt text
  • Insufficient colour contrast
  • Unlabelled form fields
  • Buttons / links with no accessible name
  • Broken ARIA roles & states
  • Heading & landmark structure

Still needs a human

  • Whether alt text is meaningful
  • Logical reading order
  • Keyboard flows in custom widgets
  • Screen-reader experience

67 rules — WCAG 2.1 AA + Best Practices + ARIA. It's a floor that never drops, not a certificate of full conformance.

  • Framework-agnostic — React, Next.js, Vue, Angular, Svelte, Astro, a static site, or a deployed preview URL.

Safety & cost

  • Open source (MIT). No licence cost.
  • Runs offline, in CI only. No page content or results leave the environment.
  • Test-only. Never shipped to end users; no production runtime.

Scoped deliberately: the optional Siteimprove cloud upload is off (no credentials), and scanning behind authentication (1FA/2FA) stays on hold pending security review.

Getting started

Adopting it

  • Pick a pilot repo — ideally one already shipping.
  • Wire it up in ~15 minutes using the step-by-step guide.
  • Start at WCAG 2.1 AA → get it green → make it a required check.
  • Then ratchet up the rule scope once you're passing.
